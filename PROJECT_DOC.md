# 服务商助手（Provider Assist）项目说明文档

## 一、项目背景

### 1.1 从售前 Agent 到服务商助手的演变

我们最初搭建了一个面向客户的**售前 Agent**（Presale Agent），让客户直接与 AI 对话完成需求调研。上线后组织了五家服务商进行内测，收到了一致的核心反馈：

> **提问质量和行业理解方面表现优秀，需求报告被评价为"准确、可交付"，但交互流程存在根本性矛盾。**

### 1.2 服务商反馈的核心问题

| 问题类型 | 具体描述 |
|---------|---------|
| **对话轮次过多** | 一轮一轮追问，客户在手机上没耐心录完那么多内容 |
| **无法现场配合** | 售前与客户面对面沟通时，不能边聊边让客户回答 AI 问题，会打断思路 |
| **先后矛盾** | 如果先和客户聊完再去填答案，很多当时确认的细节已经遗忘 |
| **第一轮卡死** | 客户回答不了第一轮问题，就到不了后续深入轮次 |
| **Demo 结构简单** | 生成的表只有 4 张，实际交付需要 10+ 张 |

### 1.3 服务商提出的新定位

五家服务商一致提出：**不要让客户直接用 AI，而是让 AI 辅助服务商做好售前工作。**

服务商真实的工作方式是：
1. 先了解客户基本情况 → 做功课、准备问题
2. 带着准备好的问题和客户面对面沟通（线下或腾讯会议）
3. 沟通完毕后，整理成方案和报价

**核心矛盾**：如果把需求引导得非常清楚 → 客户需要花很长时间回答 → 丧失耐心；如果减少问题设置 → 需求问得不够清楚。

### 1.4 解决方案：Provider Assist

我们转向"辅助服务商"这一侧，拆分为两步走：

**第一步：智能提问清单**
- 客户简短描述行业、业务和痛点
- AI 结合知识库分析，为服务商准备高质量的提问清单
- 服务商带着清单去与客户沟通（不限形式：面对面、电话、腾讯会议）

**第二步：结构化需求转化**
- 沟通完成后，服务商上传会议转写/录音文字
- AI 将非结构化沟通内容转化为：
  - (a) 客户需求分析报告
  - (b) 智能表格的字段设计
  - (c) 可直接创建的智能表格 Demo

## 三、知识库架构

### 3.1 知识库概述

知识库托管在 GitHub：**https://github.com/Serenasnchen/presale-knowledge-base**

Provider Assist 和 Presale Agent 通过 **Git Submodule** 共享同一个知识库，保证两个产品使用一致的行业数据。

### 3.2 知识库目录结构

```
presale-knowledge-base/
├── industries/          # 行业知识（17个行业）
│   ├── garment_production.json
│   ├── home_decoration.json
│   ├── b2b_sales_crm.json
│   └── ...
├── cases/               # 真实交付案例（17个案例）
│   ├── garment_production_lianwang.json
│   ├── decoration_shengxinzhu_order.json
│   └── ...
├── field_templates/     # 字段经验池（10个模板）
│   ├── garment_production_progress.json
│   ├── medical_device_manufacturing.json
│   └── ...
└── pool/                # 需求池（672条真实需求记录）
    └── demand_pool.json
```

### 3.3 四层数据架构

知识库采用四层结构设计，从宏观到微观逐层深入：

| 层级 | 目录 | 内容 | 数量 | 作用 |
|------|------|------|------|------|
| **L1 行业知识** | `industries/` | 行业定义、业务特点、常见痛点、推荐方案、AI提示 | 17 个行业 | 让 AI 理解行业背景 |
| **L2 交付案例** | `cases/` | 真实客户的需求摘要、痛点、方案架构、表结构、沟通记录 | 17 个案例 | 让 AI 学习真实交付经验 |
| **L3 字段经验池** | `field_templates/` | 经过验证的表结构设计、字段类型、自动化规则 | 10 个模板 | 让 AI 生成专业的表结构 |
| **L4 需求池** | `pool/` | 672 条真实客户需求、会议纪要、方案概述、报价 | 672 条 | 匹配相似客户需求 |

### 3.4 每层数据结构详解

#### L1 行业知识 (`industries/*.json`)

```json
{
  "meta": {
    "industry": "服装生产（样板/版房工序进度跟踪）",
    "sub_industries": ["服装生产", "样板制作", "版房管理"],
    "source": "真实案例",
    "version": "1.0"
  },
  "tags": ["服装", "生产", "样板", "版房", "工序", "进度"],
  "industry_name": "服装生产（样板/版房工序进度跟踪）",
  "content": "一、行业定义\n...\n二、核心业务特点\n...\n三、常见痛点与需求\n...\n四、推荐方案\n...\n五、AI服务专业化提示\n..."
}
```

**关键设计**：`tags` 字段用于模糊匹配，`content` 是完整的行业知识文本，包含从业务特点到方案建议的完整链路。

```json
    "industry": "服装生产",

    "client": "开平联旺",
    "service_provider": "闪鱼未来（上海）数字科技有限公司",
    "source": "服务商真实交付案例",
```

```json
{
  "meta": {
    "industry": "服装生产",
    "scene": "样板/版房工序进度跟踪",

  },
  "tables": [
    {
      "table_name": "样板进度表",
      "description": "服装样板从下单、产品信息登记到验收全流程进度、时间、耗时、状态跟踪及年度/季度/月度统计",

      "field_groups": [
        {
          "group_name": "产品信息",
          "fields": [
            {"title": "编号", "type": "TEXT", "logic": "YYMMDD-XXX格式自动生成，如260305-001"},
            {"title": "下单日期", "type": "DATE_TIME"},
            {"title": "客户", "type": "TEXT"},
            {"title": "整体进度", "type": "SINGLE_SELECT", "options": ["进行中", "已完成", "已作废"]},
            {"title": "牌子", "type": "TEXT"}
          ]
        },
        {
          "group_name": "01业务流程",
          "fields": [
            {"title": "01业务员", "type": "TEXT"},
            {"title": "01业务工作", "type": "SINGLE_SELECT", "options": ["未完成", "已完成", "✅如期", "⚠️逾期"]},

        }
      ]
    }
  ]
}
```

**关键设计**：
- `applicable_when` 列出触发该模板的关键词，让匹配算法知道何时使用此模板

- 每个字段包含 `type`（字段类型）和 `logic`（计算逻辑），AI 据此生成准确的字段设计
- `field_groups` 按业务流程分组，每组 5 个标准字段（负责人、状态、倒计时、开始期、完成期）

## 四、知识库匹配算法

### 4.1 匹配流程

当服务商输入客户的行业和业务描述后，系统执行三级匹配：

```
用户输入（行业 + 业务描述 + 痛点）
        │
        ├──→ L1 行业匹配：基于 tags + industry_name 模糊匹配
        │         → 输出：行业背景知识（content）
        │
        ├──→ L2 案例匹配：基于行业 + 场景 + 需求摘要 多维评分
        │         → 输出：Top 3 最相关交付案例
        │
        └──→ L3 模板匹配：基于行业 + 场景 + applicable_when 评分
                  → 输出：最相关的字段经验池
```

### 4.2 评分规则

**案例匹配评分**：
- 行业名称精确匹配：+5 分
- 行业关键词包含：+4 分
- 场景关键词匹配：+3 分
- 需求摘要词语命中：每词 +2 分
- 取 Top 3 输入 AI

**模板匹配评分**：
- 行业名称匹配：+5 分
- 场景关键词匹配：每词 +3 分
- applicable_when 关键词匹配：每词 +2 分
- 阈值 ≥ 4 分才采用

### 4.3 知识库与 AI 的协作方式

匹配结果作为上下文注入 AI 的 Prompt：

```
System Prompt: 你是资深智能表格售前顾问...

User Prompt:

- 行业：服装生产
- 业务描述：做样板，工序进度跟踪
- 痛点：任务状态混乱

## 行业背景知识（来自 L1）
[完整行业知识 content]

## 相关交付案例（来自 L2）
### 真实交付案例：服装生产 - 样板/版房工序进度跟踪
客户原始痛点：...
```

## 五、技术实现

### 5.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | 原生 HTML + JS | 单文件 SPA，无框架依赖 |
| 部署 | Vercel | 自动 CI/CD，Serverless Functions |
| 后端 API | Python (Vercel Functions) | 8 个 Serverless 函数 |
| AI 模型 | DeepSeek Chat | 前端直连避免超时 |
| 知识库 | GitHub + Git Submodule | 两个产品共享 |
| 企微集成 | 企微 MCP API | 创建文档、智能表格操作 |
| 数据存储 | localStorage + 企微智能表格 | 前端本地 + 后台管理表 |

### 5.2 API 接口列表

| 接口 | 功能 | 输入 | 输出 |
|------|------|------|------|
| `POST /api/question_list` | 知识库匹配 + 生成提问清单 Prompt | 行业、业务描述、痛点 | system_prompt + user_prompt |
| `POST /api/match` | 纯知识库匹配（Step2用） | 行业、方向 | 行业知识 + 案例 + 模板 |
| `POST /api/create` | 创建企微智能表格 Demo | 表结构 JSON Schema | 表格 URL |
| `POST /api/export_doc` | 创建企微文档 | 标题 + Markdown 内容 | 文档 URL |
| `POST /api/report` | 上报数据到平台管理表 | 客户数据 + 状态 + 链接 | record_id |
| `POST /api/upload` | 文件上传解析 | Word/PDF 文件 | 解析后文本 |
| `POST /api/chat` | DeepSeek 代理（备用） | prompt | AI 响应 |
| `POST /api/analyze` | 深度分析（备用） | 数据 | 分析结果 |

### 5.3 DeepSeek 直连方案

为解决 Vercel Serverless 60 秒超时限制，AI 推理改为前端直接调用 DeepSeek API：

```
前端 → /api/question_list（<5s，仅做知识库匹配，返回prompt）
前端 → DeepSeek API（直连，无超时限制，等待 AI 生成完毕）
```

### 5.4 知识库同步机制

```
knowledge/（本地源仓库）
    │
    ├── git push → GitHub: presale-knowledge-base
    │
    ├── git submodule → provider-assist/knowledge/
    │                    (Vercel 部署时自动拉取)
    │
    └── git submodule → presale-agent2/knowledge/
                         (同一份数据)
```

本地开发时可用 `sync_knowledge.sh` 一键同步到各项目目录。

## 七、与 Presale Agent 的关系

| 维度 | Presale Agent（原） | Provider Assist（新） |
|------|-------------------|---------------------|
| **服务对象** | 直接面向客户 | 辅助服务商 |
| **交互方式** | 多轮 AI 对话 | 两步式（清单 + 转化） |
| **知识库** | 同一个 presale-knowledge-base | 同一个 presale-knowledge-base |
| **核心产出** | 需求报告 | 提问清单 + 需求报告 + Demo |
| **部署** | Vercel (presale-agent) | Vercel (provider-assist) |
| **GitHub** | Serenasnchen/presale-agent2 | Serenasnchen/provider-assist |
| **状态** | 内测结束，保留运行 | 当前主力迭代 |

**两者共享同一个 GitHub 知识库**，通过 Git Submodule 引入，保证行业知识、案例数据、字段模板的一致性。更新知识库只需要在源仓库 push，两个产品部署时自动获取最新版本。
